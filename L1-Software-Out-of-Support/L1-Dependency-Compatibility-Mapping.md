## 1) Why this activity (value and decision)
Upgrade paths in legacy codebases are rarely straightforward. Frameworks, libraries and runtime versions create a dependency graph where upgrading one component may require upgrading others first, or may introduce breaking changes downstream. Without a clear map, teams either stall or attempt upgrades in the wrong order and waste effort on rework.

This activity produces a sequenced compatibility map so the team knows which upgrades are safe, which have prerequisites, and where breaking changes will need code remediation. It directly enables the Upgrade Plan and Exemplar PR activities.

Decision enabled: approve the upgrade sequence and identify early blockers before committing engineering time.


---

## 2) What we will do (scope and steps)
Description: Analyse repository and SBOM to determine safe upgrade paths and prerequisites.

Sub tasks:
1. Gather the SBOM output and shortlist from the Extract SBOM activity. If the SBOM activity has not yet run, flag the dependency and defer.
2. For each shortlisted component, use AI to retrieve the target version's release notes, migration guides, and changelog entries. Prompt the AI to identify: (a) breaking API changes, (b) deprecated features removed, (c) minimum runtime or platform version required.
3. Map the dependency graph: identify which components depend on each other and where upgrading component A requires component B to be upgraded first. Represent this as a simple ordered list or diagram.
4. For each upgrade step, note: estimated code change scope (trivial/minor/significant), test areas likely affected, and any configuration changes needed.
5. Flag any circular or conflicting dependencies that prevent a clean upgrade path.
6. Produce a one-page compatibility map showing the recommended upgrade sequence, prerequisites at each step, and a confidence tag (High/Medium/Low) per step.
7. Identify 2-3 dependencies that are good candidates for the Exemplar Upgrade PRs activity (low-risk, high-value, clear path).
8. Log time spent (start/end timestamps) for P1 measurement.

> **Sequencing:** depends on Extract SBOM output. Feeds into Upgrade Plan and Breaking Change Notes and Exemplar Upgrade PRs. Run in Week 2 (early Execute phase).

> **Out of scope:**
> - Executing any upgrades (see Exemplar Upgrade PRs).
> - Writing migration code or refactoring beyond noting expected changes.
> - Upgrading transitive dependencies that are not in the shortlist.

---

## 3) How AI is used (options and modes)
- **Retrieve and ground:** pull release notes, changelogs, and migration guides for each shortlisted component to establish the factual basis for compatibility analysis.
- **Analyse and reason:** map version ranges against the current codebase versions, detect breaking API changes, flag deprecated features, and identify prerequisite upgrades in the dependency graph.
- **Generate:** draft the compatibility map document, produce the ordered upgrade sequence, and write summary notes for each step including estimated change scope.
- **Human in the loop:** an engineer with knowledge of the codebase validates the dependency graph, confirms the upgrade order, and adjusts confidence tags based on domain knowledge.

## 6) Timebox
Suggested: 1h for a repository with fewer than 20 direct dependencies; 1.5h for larger or multi-module repositories. Schedule in Week 2 (early Execute phase), after the Extract SBOM activity has completed.


---

## 7) Inputs and data sources
- SBOM file and component shortlist from the Extract SBOM activity.
- Repository source code and lockfiles (for verifying actual usage of flagged APIs).
- Release notes, changelogs, and migration guides for shortlisted components (AI will retrieve; confirm internet access or provide cached copies).
- Any existing upgrade notes or ADRs the department already holds.
- If unavailable: state assumptions (e.g. "no prior upgrade notes exist; mapping based solely on SBOM and public release notes") and tag confidence as Medium or Low.


---

## 8) Outputs and artefacts
- One-page compatibility map (Markdown or spreadsheet): ordered upgrade sequence with prerequisites, estimated code-change scope per step, and a confidence tag (High/Medium/Low) for each.
- Prerequisite list: platform or runtime upgrades that must happen before any library upgrades can proceed (e.g. "Node.js must move from 14 to 18 before upgrading React").
- Shortlist of 2-3 exemplar PR candidates with rationale.
- Time log entry for P1.

Audience: Solution Architect, Tech Lead, department engineering team. The compatibility map may also inform the Upgrade Plan and Breaking Change Notes activity.

## 10) Risks and controls

| Risk | Mitigation |
|---|---|
| **Stale AI training data**: the AI's knowledge of recent releases may lag behind actual vendor announcements | prompt the AI to cite sources; cross-check the top 3-5 flagged breaking changes against official release notes |
| **Missed transitive dependencies**: the SBOM may not capture all transitive dependency conflicts | run a dependency tree command (e.g. npm ls, mvn dependency:tree) as a supplementary check and compare against the SBOM |
| **Overconfident upgrade sequence**: AI may propose an order that overlooks runtime or build-tool constraints | reviewer validates each prerequisite step against the actual build pipeline before the map is finalised |
| **Scope creep into remediation**: the temptation to start fixing discovered issues during this activity | log findings and defer to the Exemplar Upgrade PRs and Upgrade Plan activities |

## 11) Review and definition of done
Done when all of the following are true:
- [ ] Compatibility map covers all shortlisted components with an ordered upgrade sequence.
- [ ] Each step has a confidence tag (High/Medium/Low) and estimated code-change scope.
- [ ] Prerequisites for platform or runtime upgrades are explicitly listed.
- [ ] Exemplar PR candidates are identified and agreed with the reviewer.
- [ ] Solution Architect or Tech Lead has reviewed and approved the map.
- [ ] Time log entry is recorded for P1.
- [ ] Evidence Log and Evaluation Scorecard are updated.
- [ ] Decision Log entry added if a sequence change or scope adjustment is recommended.

## 12) Playbook contribution
- **Where AI helped**: time saving on release-note analysis and dependency graph reasoning versus manual triage; quality of the compatibility map compared to a manually drafted equivalent.
- **Prerequisites to document**: SBOM availability, lockfile completeness, internet access for release notes.
- **Limits and risks to document**: any cases where AI proposed an incorrect upgrade order or missed a prerequisite.
- **Reusable assets**: compatibility map template, AI analysis prompt, dependency-tree generation command.

Pattern candidates:
- **"SBOM-to-map pipeline"**: feeding SBOM output directly into an AI prompt to generate the compatibility map is faster and more complete than starting from scratch. Record the time saving.
- **"Confidence-tagged sequencing"**: tagging each upgrade step with a confidence level helps teams prioritise which steps to validate first.

Anti-pattern candidates:
- **"Mapping without lockfiles"**: running compatibility analysis on manifest-only SBOMs produces incomplete maps. Always check for lockfile availability first.
- **"AI-only upgrade order"**: accepting the AI's proposed sequence without engineer validation risks missing build-tool or runtime constraints that the AI cannot see.
